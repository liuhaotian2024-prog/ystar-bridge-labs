# Y*gov Setup

Set up governance for this project in 3 steps.

## Step 1: Check installation

```bash
python3 -c "import ystar; print('Y*gov', ystar.__version__, '— ready')" 2>/dev/null \
  || echo "Y*gov not installed — run: pip install ystar"
```

If not installed, tell the user to run `pip install ystar` first, then re-run this skill.

## Step 2: Analyze the current project

```bash
# Get a sense of the project structure
ls -la
echo "---"
# Check if AGENTS.md already exists
[ -f AGENTS.md ] && echo "AGENTS.md exists:" && cat AGENTS.md || echo "No AGENTS.md found"
```

## Step 3: Generate AGENTS.md

Based on the project structure, create a sensible `AGENTS.md`. Use the template below as a starting point, then adapt it to the actual project:

```markdown
# AGENTS.md — Y*gov Governance Contract
# Auto-generated by Y*gov setup. Edit as needed.
# Documentation: https://github.com/liuhaotian2024-prog/Y-star-gov

---

## Absolute denials (all agents)

- /etc
- /root
- /production
- .env
- .aws/credentials
- ~/.ssh

## Denied commands (all agents)

- rm -rf
- sudo
- DROP TABLE
- DELETE FROM
- git push --force

## Allowed paths (restrict writes to project directory)

- ./
- /tmp

## Multi-agent delegation rules

### Orchestrator
- May spawn: any subagent defined in .claude/agents/
- May NOT access production systems directly
- All spawned subagents inherit these restrictions

### Subagents (general)
- May NOT spawn other subagents
- Must complete assigned task within 300 seconds
- Must report status before accessing new paths

## Obligations (SLA)

- task_acknowledgement: 60 seconds
- task_completion: 600 seconds
- status_update: 300 seconds
```

After creating the file, confirm with the user and suggest:

1. **Review the rules** — edit AGENTS.md to match your project's actual requirements
2. **Install the hook** — `ystar hook-install` to register the PreToolUse hook
3. **Verify setup** — `ystar doctor` to confirm everything is working

## Notes on writing good governance rules

- **Denied paths**: think about what your agents should never touch (production, credentials, system files)
- **Allowed paths**: leave empty if you want to allow all paths, or restrict to `./` for safety
- **Obligations**: set realistic SLAs — too tight will cause unnecessary blocks
- **Subagent rules**: the most important section for multi-agent workflows
