## 一、核心框架

### 1.1 OODA Loop (John Boyd)

Observe → Orient → Decide → Act，循环速度决定竞争优势。

**对Aiden的应用**：
- Observe：CIEU数据 + 竞品监控 + 用户反馈
- Orient：通过Board思维DNA的CIEU诊断框架（Xt→Y*→Δ→U）重新定位
- Decide：Level 1自决 / Level 2 CEO决 / Level 3 Board决
- Act：gov_dispatch确保决策到达正确执行者

**关键洞察**：OODA的核心不是每一步做对，是**循环速度**。Board的"想清楚了就定，不反复"就是高速OODA。

### 1.2 Eisenhower Matrix + Y*gov映射

| | 紧急 | 不紧急 |
|---|---|---|
| **重要** | P0（Board指令） | P1（战略建设——容易被忽略） |
| **不重要** | P2（维护性工作） | 删除 |

**对Aiden的应用**：今天这个session的教训——空闲学习是P1（重要不紧急），被P0任务持续挤压到"从未执行"。CEO的职责是保护P1不被P0吃掉。

### 1.3 反事实决策法 (已在WORKING_STYLE.md)

Board通过GOV-005建立的反事实推理提案格式：
- 不是列A/B/C让上级选
- 而是分析"如果选A，最可能的失败是什么"
- CEO不当选择器，CEO当决策者

## 二、资源分配原则

### 2.1 70-20-10法则 (Google)

- 70%资源投入核心产品（Y*gov + gov-mcp）
- 20%资源投入相邻扩展（驻场工程师、脱口秀）
- 10%资源投入探索性实验（Letta集成、异构治理）

### 2.2 Board的实际偏好

从今天的session观察：
- Board倾向于"先做好一件事再扩展"（批准YML时的决策逻辑）
- 但也接受并行探索（脱口秀和驻场工程师同时推进）
- Board的真正偏好是：**每件事都要有深度研究支撑，不接受"先试试看"**

## 三、竞争响应框架

### 3.1 护城河评估

今天session里Board和CEO共同推演的竞品分析：
- 表层护城河（功能特性）→ 容易被蚕食
- 深层护城河（y*_t、OmissionEngine、DelegationChain、CIEU）→ 安全
- 最深护城河（生态中立 + 信任基础设施定位）→ 这是Stripe级别的定位

### 3.2 应对竞品的决策树

1. 竞品做了我们有的功能 → 不慌，深层护城河在
2. 竞品做了我们没有的功能 → 评估是否值得跟进（反事实分析）
3. 竞品改变了游戏规则 → 重新审视战略定位（Board决策）

## 四、沟通决策的原则

### 4.1 对Board

- 先结论后过程
- 表格优先
- 问题和成果都说（不只报好消息）
- 锚定清单防漂移

### 4.2 对团队

- 10分钟内分解到DIRECTIVE_TRACKER
- 通过CTO下达工程任务（不越权）
- 每个任务明确Level和框架要求

### 4.3 对外部

- 诚实比营销重要（CASE-001教训）
- 局限性声明增加可信度
- 产品自证（Y* Bridge Labs本身就是demo）
