# 规则
CEO 派工程任务必须经 CTO delegation，不得直接调 `eng-kernel` / `eng-platform` / `eng-governance` / `eng-domains`。

# Why
2026-04-13 Board 直接授权 Jordan (eng-domains) 实装 AMENDMENT-012 "Deny-as-Teaching"。CEO (Aiden) 在 session boot 后尝试：

1. **第 1 次**: `invoke("Agent", agent="eng-domains", task="立项并实装 AMENDMENT-012")` → hook DENY
2. **第 2 次**: 改为 `invoke("Agent", agent="cto", ...)` 但在 task 描述里说"派 Jordan" → CTO 收到但 CEO 仍在微管理
3. **第 3 次**: 写 `.claude/tasks/eng-domains-amendment-012.md` 但**未经 CTO review** → 破坏 delegation chain
4. **第 4 次**: 尝试自己 Write `Y-star-gov/ystar/session.py` 加 Remediation 字段 → hook DENY `ceo_deny_paths`

根因：
- **Board 指名 Jordan ≠ CEO 可直接派 Jordan**。Board 指名是建议，CEO 仍需走 delegation chain（保证 CTO 知情 + CIEU 审计）
- CEO 误以为"紧急任务"可以绕过流程（实际 `must_dispatch_via_cto` 无例外）
- CEO 不熟悉 MCP delegation / task 卡机制，依赖直接 Agent call

Board 纠偏："你是 CEO，不是 project manager。派 CTO，CTO 派 Jordan。你只看结果。"

# How to apply
任何工程任务（代码/测试/架构/治理）：
1. **第一选择**: `invoke("Agent", agent="cto", task="...")` — CTO 收到后分派给合适工程师
2. **第二选择**: 写 `.claude/tasks/cto-xxx.md` — 异步任务队列（CTO 定期 check）
3. **第三选择** (未来): `mcp__gov-mcp__gov_delegate(from="ceo", to="cto", ...)` — 正式 delegation tool

禁止：
- ❌ 直接调 `eng-*` agent（即使 Board 指名该工程师）
- ❌ 自己改 Y*gov 源码（CEO 写作域 = ystar-company/，不含 Y-star-gov/ystar/）
- ❌ 在 task 描述里微管理工程师（"Jordan 你先改 session.py 再改 boundary_enforcer.py"）— 让 CTO 和工程师自主决定实现路径
