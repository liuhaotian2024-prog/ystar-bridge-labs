## Key Concepts Learned

### 1. The Ladder of Causation (Three Rungs)

Pearl defines three levels of causal reasoning:

**Rung 1: Association (Seeing)**
- "What is?" — observational queries
- P(Y | X) — probability of Y given we observe X
- Example: "If I see an agent violate a rule, what is the probability it will violate again?"
- Y* equivalent: CIEU CallRecord historical analysis

**Rung 2: Intervention (Doing)**
- "What if I do?" — interventional queries
- P(Y | do(X)) — probability of Y if we actively set X (not just observe)
- Example: "If I tighten the constraint, what happens to violation rate?"
- Y* equivalent: InterventionEngine active pulses

**Rung 3: Counterfactuals (Imagining)**
- "What if I had done?" — retrospective reasoning
- P(Y_x | X'=x', Y'=y') — what would Y be if X were x, given we observed X'=x' and Y'=y'
- Example: "Would this violation have occurred if the constraint were tighter?"
- Y* equivalent: Counterfactual replay in causal_engine.py

### 2. Do-Calculus Rules

Pearl's three rules for converting observational distributions to interventional:

**Rule 1 (Ignoring observations)**:
P(y | do(x), z, w) = P(y | do(x), w) if (Y ⊥ Z | X, W) in G_X̄

**Rule 2 (Action/observation exchange)**:
P(y | do(x), do(z), w) = P(y | do(x), z, w) if (Y ⊥ Z | X, W) in G_X̄Z

**Rule 3 (Ignoring interventions)**:
P(y | do(x), do(z), w) = P(y | do(x), w) if (Y ⊥ Z | X, W) in G_X̄Z(W)

These rules allow computing causal effects from purely observational data when certain conditions hold.

### 3. Counterfactual Identifiability

A counterfactual P(Y_x | X'=x', Y'=y') is identifiable if it can be computed from:
1. The observational distribution P(V)
2. The causal graph G
3. Knowledge of functional relationships

Not all counterfactuals are identifiable (fundamental limitation).

## Application to Y*gov

### Current Y* Architecture Maps to Pearl's Ladder

1. **Rung 1: YStarLoop + metalearning**
   - `learn(history, contract)` analyzes observational data
   - Statistical patterns in CallRecord history
   - This is pure association

2. **Rung 2: InterventionEngine**
   - `gate_check()` actively blocks actions (intervention)
   - do(block_spawn) vs observe(spawn_blocked)
   - This is true causal intervention

3. **Rung 3: causal_engine.py counterfactual replay**
   - "Would this violation have happened if constraint X were different?"
   - Replay history with modified contract
   - This is counterfactual reasoning

**Y* is already operating at all three rungs of Pearl's ladder.**

### Gap Identified: Do-Calculus Not Formalized

We do counterfactual replay, but we don't use Pearl's do-calculus rules to:
- Identify which counterfactuals are identifiable from CIEU data
- Compute interventional distributions from purely observational CIEU records
- Prove which governance suggestions are causally valid vs just correlated

### Experiment 2: Identify Non-Identifiable Counterfactuals

Question: "Would this agent have complied if we had sent a reminder earlier?"

To answer, need:
- Observational data: CIEU records of reminder timing vs compliance
- Causal graph: Reminder → Agent_State → Compliance
- Functional form: How does agent internal state respond to reminders?

**Problem**: Agent internal state is unobservable (black box).

**Pearl's answer**: This counterfactual is NOT identifiable without assumptions about agent internals.

**Y* implication**: We can only compute identifiable counterfactuals. Must add identifiability checking to causal_engine.py.

### Experiment 3: Simulate Do-Calculus Rule 1

Toy example:

```python
# Observational data (simulated)
P_violation_given_tight_and_urgent = 0.3
P_violation_given_tight = 0.2
P_violation_given_urgent = 0.4

# Causal graph: Urgent → Tight (urgent tasks get tight constraints)
#               Tight → Violation
#               Urgent → Violation

# Question: P(violation | do(Tight=True)) = ?
# Cannot just use P(violation | Tight=True observed) = 0.2
# because tight constraints are confounded with urgent tasks

# Pearl Rule 1: Remove confounding by conditioning on parent (Urgent)
# P(V | do(T=1)) = Σ_u P(V | T=1, U=u) * P(U=u)
# Assuming P(Urgent=1) = 0.5:
P_do_tight = 0.3 * 0.5 + 0.15 * 0.5  # (rough estimate)
# = 0.225

# Interpretation: Active intervention (tightening) has 22.5% violation rate
# vs 20% when tight is merely observed (confounded with urgent=True)
```

**Learning**: Observational data underestimates intervention effect when there's confounding.

## Open Questions

1. **Causal sufficiency**: Does Y* CIEU schema capture all relevant confounders?
   - CIEU records: tool_name, params, decision, violations, drift
   - Missing: agent internal state, task context, external events
   - Need to add context fields to CallRecord?

2. **Structural equations**: How do we learn functional forms f(X) from CIEU data?
   - Pearl assumes F = {f₁, f₂, ...} are known
   - In Y*, we only have statistical patterns
   - Can we use neural causal models? (Bengio et al. 2019)

3. **Time-varying causation**: Causal graph changes over time as agents learn
   - C → A edge strength increases as agent internalizes constraints
   - How to handle non-stationary causal graphs?
   - Pearl's framework assumes static graphs

4. **Multi-agent causal interference**: Agent A's behavior affects Agent B's context
   - Violates SUTVA (Stable Unit Treatment Value Assumption)
   - Pearl's do-calculus doesn't handle interference
   - Need to read Hudgens & Halloran (2008) on causal inference with interference
