## 方法论1：跨模型交叉审计法

**发现过程：** 老大用ChatGPT独立审计Y*gov代码，发现了我们团队（Claude）忽略的问题。

**方法：**
1. 用不同AI模型对同一代码库做独立分析
2. 对比两个模型的结论差异
3. 差异点就是盲区——最有价值的发现

**实证：**
- ChatGPT发现Path A定义太窄（我们只看PathAAgent，ChatGPT看到7个组件的生态系统）
- ChatGPT发现contract legitimacy decay在文章里有但代码里没有
- ChatGPT三轮审计共发现12个工程问题，全部修复

**适用场景：** 任何重大功能完成后，用另一个AI模型做独立审计。

## 方法论2：Pearl因果层级作为架构验证工具

**发现过程：** 老大提出CIEU五元组本身就是Pearl启发的，能否用Pearl来验证我们的因果假设。

**方法：**
1. 定义SCM变量（W,O,H,S）
2. 从生产数据拟合参数
3. 参数化自举生成合成数据
4. PC算法发现因果结构
5. 与手工指定的DAG对比（SHD度量）
6. 时间顺序作为背景知识解决Markov等价类歧义

**实证：**
- PC算法骨架4/4完美匹配
- 时间顺序约束后SHD=0（完美匹配）
- LiNGAM独立确认2/4方向
- 882条CIEU记录→1个cycle级观测（数据管道很重要）

**关键教训：** 原始记录数≠SCM数据点数。必须做cycle级聚合。

**适用场景：** 任何基于历史数据验证系统因果假设的场景。

## 方法论3：自治理审计法（吃自己的狗粮）

**发现过程：** CTO后台超时25分钟无人知道——我们造了治理系统却没用在自己身上。

**方法：**
1. 列出产品的所有治理机制（Y*gov有12个）
2. 逐个检查：团队自己是否在用？
3. 落实率=在用数/总数
4. 差距=有代码能力但团队未使用的机制

**实证：**
- 审计发现12个机制只有2个在用（16.7%）
- enforce()静默降级bug：EventType.SHELL_EXEC不存在，except:pass吞掉错误
- 从安装以来，完整enforce()路径从未运行过
- 修复后：12/12全部写入宪法级规则

**关键教训：** "功能存在"≠"功能在用"。自审计是发现这种差距的唯一方法。

**适用场景：** 任何产品团队在发布前做self-dogfooding检查。

## 方法论4：架构冻结法（一底座三条线）

**发现过程：** 顾问建议先定结构再加功能，否则功能越多结构越混。

**方法：**
1. 全代码库审计，标记每个模块归属哪一层
2. 写ARCHITECTURE_FREEZE文档，明确5层+调用方向规则
3. 目录重构：代码目录与架构叙事一致
4. Layer声明：每个模块头部标记`# Layer: Foundation/Path A/Path B/Bridge/Intent Compilation`
5. 架构测试：test_architecture.py检查跨层import禁令
6. 准入规则：新模块必须声明归属
7. 变更门槛：改顶层结构需Board审批

**实证：**
- 从94个模块散落 → 5层清晰归属
- Path A/B从module_graph/移到独立目录
- CausalEngine从module_graph/移到governance/（底座归属纠正）
- 267→304→331测试逐步加固

**关键教训：** 先冻结结构，再补功能。否则"功能完成"时结构已经混乱。

**适用场景：** 任何代码库从原型期进入产品期的关键转折点。

## 方法论5：功能完成三约束（底座优先/IO无关/少硬编码）

**发现过程：** 顾问在functional-complete阶段提出三条总约束。

**方法：**
对每个新功能或修改问三个问题：
1. **底座优先：** 这个功能是走底座统一入口，还是在上层做旁路？
2. **IO无关：** 这个功能是绑死某个CLI/runtime/格式，还是与输入输出无关？
3. **少硬编码：** 这个参数是写死在函数里，还是来自policy object/配置/自适应？

**实证：**
- Path A forbidden_paths写死 → 迁到PathAPolicy
- Path A/B直读constitution → 迁到ConstitutionProvider
- module:前缀拼接写死 → 迁到scope_encoding.py
- GovernanceLoop直调_prefill() → 迁到compiler.compile_source()

**关键教训：** 硬编码在Pearl L3实现后应该自然消亡——自适应参数需要策略对象作为接口。

**适用场景：** 任何从"能跑"到"可配置/可扩展"的工程升级。

## 方法论6：不作为治理8层落实法

**发现过程：** 对照OmissionEngine代码能力，发现团队0%落实。

**8层：**
1. 义务8状态生命周期
2. 两阶段超时（SOFT 5min → HARD 15min）
3. 7种不作为类型（未委托/未确认/未更新/未发布/未通知/未升级/未关闭）
4. 5步升级策略（REMINDER→VIOLATION→ESCALATE→DENY_CLOSURE→SEVERITY_UPGRADE）
5. 行动触发式检测（agent下一次动作触发义务扫描）
6. 自动义务创建（指令→义务+deadline）
7. 主动干预（3次失败→暂停，HARD_OVERDUE→强制先做）
8. 禁止假完成（关闭前检查所有子义务）

**适用场景：** 任何团队管理中需要追踪"该做但没做"的场景。

## 方法论7：产品文档对称法

**发现过程：** 顾问发现产品包只有Path A演示，没有Path B——这会让外界误解产品边界。

**方法：**
产品级文档必须覆盖系统的每一条主线，不能偏向"最酷的概念"而忽略"最实用的能力"。

**正确结构：**
1. 一页总架构图（覆盖全系统）
2. 每条线各一个演示包（Path A + Path B + 联动）
3. 分层集成指南（不同用户不同起点）

**关键教训：** Path A是学术旗帜，Path B是商业入口。只讲Path A = 只展示一半产品。

## 方法论8：版本发布验证链

**发现过程：** 顾问下载的GitHub ZIP是旧版——我们push了但缓存导致不一致。

**方法：**
每次push后CTO必须验证：
1. git rev-parse HEAD == origin/main
2. raw.githubusercontent.com能拿到最新文件
3. 版本号一致（__init__.py = pyproject.toml = README）
4. PyPI已上传（如有版本bump）
5. GitHub Release已创建（如有里程碑）

**关键教训：** "push了"≠"对方能看到"。验证是CTO的义务，不是老大的。

## 方法论9：金金委托协议（跨模型成本优化）

**发现过程：** 团队直接用Claude Opus做研究任务，浪费token。

**方法：**
1. 研究/收集任务先派给金金（MiniMax，便宜一个数量级）
2. 发任务后1分钟×3次查收回复
3. 金金结果需要验证后才能用于决策
4. 通信记录写入session log

**实证：**
- 4个平台研究、Pearl论文调研、公司管理研究全部由金金完成
- 节省了大量Claude Opus token
- 金金回复速度通常在2-3分钟

**关键教训：** 不是所有任务都需要最强模型。成本纪律是AI团队的基本功。

## 方法论11：Skill自主增强策略

**发现过程：** 老大提出团队不应该等待skill被推荐，而应该主动搜索、评估、安装。

**方法：**
1. 每个岗位按"岗位能力最大化"原则主动搜索skill
2. 遇到新任务先搜索是否有匹配skill
3. 评估匹配度：skill价值 = 任务匹配度 × 复用频率
4. 匹配度高就自行安装，不需要审批
5. 持续进化——不是一次性配置

**本质：** 这是Y*gov"自适应参数"理念的团队版。
跟Pearl L3一样：从经验中发现什么有效，自动扩展能力面。

**适用场景：** 任何AI团队的持续能力建设。不限于skill，也适用于知识库、工具链、工作流的持续优化。
