## §0 TL;DR — The Second Architecture Blind Spot

**Problem:** Y*gov governance codifies "deny" rules but not "allow" rules. Agents know what they *cannot* do only after hitting a wall, but never know what they *can* do proactively. This creates a pathological loop: try → deny → guess workaround → deny again → escalate/defer.

**Evidence:** Current session CEO hit **276 intervention_gate:deny events** (CIEU audit, event_type `intervention_gate:deny`). Every deny returned only `block_msg`, no `your_envelope` or `alternative_paths_in_envelope`.

**Solution:** Three-pronged capability visibility mechanism:
1. **Positive capability schema** (`role_capabilities.yaml` per role): explicit `can_write_paths`, `can_read_paths`, `can_call_tools`, `can_dispatch_to`, `escape_hatches`
2. **Boot pack Category 1.5** ("Capability Envelope Summary"): agent sees their envelope at session start
3. **Hook deny enrichment**: every deny returns `{decision, your_envelope_diff, alternative_paths_in_envelope}` instead of just `block_msg`

**Relationship to AMENDMENT-016:** 016 = transport layer (rule sync); 017 = payload layer (rule schema must contain allow half + visibility forcing function).

**Board Decision Required:** Approve 48h parallel implementation with 016 (one task split: Platform Engineer owns hook deny enrichment, Governance Engineer owns capability YAML + boot pack injection).

## §2 Proposal — Three-Pronged Capability Visibility

### 2.1 Positive Capability Schema (New Artifact)

**File:** `governance/role_capabilities.yaml` (single source, generated by AMENDMENT-016 sync engine)

**Schema:**
```yaml
roles:
  ceo:
    can_read_paths:
      - "**/*"  # CEO has universal read
      exceptions:
        - ".env"
        - "finance/sensitive/*"
    can_write_paths:
      - "reports/**"
      - "content/**"
      - ".claude/tasks/**"
      - "DIRECTIVE_TRACKER.md"
      - "memory/session_handoff.md"
    cannot_write_paths:  # explicit deny list
      - "DISPATCH.md"  # owned by Secretary
      - ".ystar_session.json"  # owned by Secretary
      - "src/**"  # owned by CTO
    can_call_tools:
      - "Read"
      - "Write"
      - "Bash"
      - "Agent"
      - "Skill"
      - "mcp__gov-mcp__gov_*"  # all GOV MCP tools
    can_dispatch_to:
      - "cto"
      - "cmo"
      - "cso"
      - "cfo"
      - "secretary"
    must_dispatch_via:  # mandatory delegation rules
      code_tasks: "cto"  # CEO never writes code directly
      governance_updates: "secretary"  # CEO never updates session.json directly
    default_actions:  # what to do when blocked
      write_denied: "write task card to .claude/tasks/{target_role}_*.md"
      read_denied: "delegate via Agent tool to secretary for read-only access"
      escalation_path: "Board (via direct message in conversation)"
    escape_hatches:  # valid workarounds by category
      priority_update: "write to .claude/tasks/secretary_priority_update.md"
      code_fix_needed: "write to .claude/tasks/cto_*.md or invoke Agent tool with cto"
      content_approval: "draft in content/drafts/, ping Board in next report"

  cto:
    can_read_paths:
      - "src/**"
      - "tests/**"
      - "products/ystar-gov/**"
      - ".github/**"
      - ".claude/tasks/**"  # CTO can read task assignments
      - "reports/**"  # CTO can read all reports
    can_write_paths:
      - "src/**"
      - "tests/**"
      - "products/ystar-gov/**"
      - ".github/workflows/**"
      - ".claude/tasks/eng-*.md"  # CTO can write engineer task cards
      - "reports/proposals/charter_amendment_*.md"  # CTO can propose amendments
      - "reports/cto_fix_log.md"
    cannot_write_paths:
      - ".env"
      - "finance/**"
      - "sales/**"
      - ".ystar_session.json"
      - ".claude/agents/**"  # cannot modify own role definition
    can_call_tools:
      - "Read"
      - "Write"
      - "Edit"
      - "Bash"
      - "Grep"
      - "Glob"
      - "Agent"  # CTO can spawn engineer sub-agents
      - "mcp__gov-mcp__gov_*"
    can_dispatch_to:
      - "eng-kernel"
      - "eng-governance"
      - "eng-platform"
      - "eng-domains"
    must_dispatch_via:
      ystar_source_changes: "eng-{kernel|governance|platform|domains} based on module"
      test_failures: "eng-{responsible_module}"
    default_actions:
      write_denied: "write task card to .claude/tasks/eng-{module}-*.md"
      ystar_install_issue: "spawn eng-platform sub-agent for hook/adapter debugging"
      test_failures: "spawn responsible module engineer for fix"
    escape_hatches:
      emergency_cto_code_write: "only if all 4 engineers unavailable AND P0 blocker"
      cross_module_review: "CTO can read all src/ but must delegate writes"

  secretary:
    can_read_paths:
      - "**/*"  # Secretary has universal read (curated role)
    can_write_paths:
      - ".ystar_session.json"
      - ".claude/agents/**"
      - "AGENTS.md"  # Secretary curates governance contract
      - "DISPATCH.md"
      - "knowledge/secretary/**"
      - "memory/**"
      - "reports/boot_packages/**"
    cannot_write_paths:
      - ".env"
      - "src/**"  # Secretary doesn't write code
      - "finance/sensitive/**"
    can_call_tools:
      - "Read"
      - "Write"
      - "Edit"
      - "Bash"
      - "Grep"
      - "Glob"
      - "mcp__gov-mcp__gov_secretary_curate_trigger"
      - "mcp__gov-mcp__gov_contract_*"
    can_dispatch_to: []  # Secretary does not dispatch, only responds to requests
    must_dispatch_via: {}
    default_actions:
      governance_conflict: "escalate to Board with CIEU evidence"
      rule_update_request: "run curate workflow, update AGENTS.md + session.json + sync"
    escape_hatches:
      board_directive_override: "Board can override any rule; Secretary implements via curate"

# Future roles: eng-kernel, eng-governance, eng-platform, eng-domains, cmo, cso, cfo
# Each follows same schema
```

**Key design decisions:**
1. **Whitelist + blacklist**: Both `can_write_paths` and `cannot_write_paths` exist. Deny takes precedence.
2. **Escape hatches are first-class**: Every role has explicit "when blocked, do THIS" paths.
3. **Machine-readable**: YAML allows programmatic query by GOV MCP tools.
4. **SSoT**: This file is generated by AMENDMENT-016 sync engine, not hand-edited.

### 2.2 Boot Pack Category 1.5 — Capability Envelope Summary

**Change:** Add new category to boot package between Category 1 (Session Identity) and Category 2 (Contract Constraints).

**Example output** (for CEO role):
```
══════════════════════════════════════════════════════════════════════
Category 1.5: YOUR CAPABILITY ENVELOPE
══════════════════════════════════════════════════════════════════════

✅ YOU CAN:
  Read:   All files except .env, finance/sensitive/*
  Write:  reports/**, content/**, .claude/tasks/**, DIRECTIVE_TRACKER.md, memory/session_handoff.md
  Tools:  Read, Write, Bash, Agent, Skill, all mcp__gov-mcp__gov_* tools
  Dispatch: cto, cmo, cso, cfo, secretary

❌ YOU CANNOT:
  Write: DISPATCH.md (owned by Secretary), .ystar_session.json (owned by Secretary), src/** (owned by CTO)
  Code tasks: Must delegate to CTO (you are CEO, not developer)

🔓 WHEN BLOCKED:
  Write denied → write task card to .claude/tasks/{target_role}_*.md
  Priority update needed → write to .claude/tasks/secretary_priority_update.md
  Code fix needed → invoke Agent tool with cto or write to .claude/tasks/cto_*.md

Your complete capability schema: governance/role_capabilities.yaml (role: ceo)
Query anytime: mcp__gov-mcp__gov_check --actor ceo --show-envelope
══════════════════════════════════════════════════════════════════════
```

**Forcing function:** Agent sees this at every session start. No excuse for "I didn't know I could do X."

### 2.3 Hook Deny Enrichment — Return Envelope Diff

**Current hook deny response:**
```json
{
  "decision": "deny",
  "block_msg": "Cannot write to DISPATCH.md (restricted_write_paths)"
}
```

**Proposed enriched response:**
```json
{
  "decision": "deny",
  "block_msg": "Cannot write to DISPATCH.md (restricted_write_paths)",
  "your_envelope": {
    "can_write": ["reports/**", "content/**", ".claude/tasks/**", "DIRECTIVE_TRACKER.md"],
    "cannot_write": ["DISPATCH.md", ".ystar_session.json", "src/**"]
  },
  "alternative_paths_in_envelope": [
    "Write task card to .claude/tasks/secretary_update_dispatch.md with instructions",
    "Delegate to secretary via Agent tool for governance file updates"
  ],
  "query_capability": "mcp__gov-mcp__gov_check --actor ceo --show-envelope"
}
```

**Implementation:** Hook adapter reads `governance/role_capabilities.yaml` (synced by AMENDMENT-016 engine), extracts `can_write_paths` + `escape_hatches.{relevant_category}`, appends to deny response.

**Enforcement:** AMENDMENT-016 rule schema v2 requires `compliance_recipe` field. Hook uses that field to populate `alternative_paths_in_envelope`.

## §5 Risks & Mitigation

### Risk 1: Whitelist too restrictive → agents blocked on valid work
**Probability:** High (first version always misses edge cases)  
**Impact:** Medium (agent defers work, Board manually approves)  
**Mitigation:**
- Default policy: **allow by default, deny explicitly**. If a path is not in `cannot_write_paths`, assume allowed.
- Add `escape_hatches.fallback: "escalate to Board with context"` as universal last resort.
- Monitor CIEU deny events daily. If a new deny pattern emerges (>5 denies on same path/role combo), treat as signal to update YAML.

### Risk 2: Capability YAML diverges from AGENTS.md
**Probability:** Medium (sync engine may fail)  
**Impact:** High (governance inconsistency)  
**Mitigation:**
- AMENDMENT-016 sync engine auto-generates `role_capabilities.yaml` from GOV MCP contracts.
- Secretary curate workflow verifies YAML ↔ AGENTS.md alignment every session.
- If divergence detected, Secretary escalates to Board.

### Risk 3: Boot pack Category 1.5 too verbose → agent ignores it
**Probability:** Low (agents read boot output by protocol)  
**Impact:** Medium (defeats purpose of visibility)  
**Mitigation:**
- Keep Category 1.5 under 15 lines.
- Use emoji prefixes (✅❌🔓) for visual anchoring.
- Add "Query anytime: {mcp_tool_command}" footer so agent knows how to re-query.

### Risk 4: CAPABILITY_REMINDER fires too often → spam
**Probability:** Low (N=50 is conservative)  
**Impact:** Low (agent ignores, but CIEU log bloat)  
**Mitigation:**
- Make N configurable via `.ystar_session.json` field: `"capability_reminder_interval": 50`.
- If Board feedback = "too noisy", bump to N=100.

## §7 Board Decision Points

This amendment requires Board approval on **3 design choices**:

### Decision 1: Default policy = allow or deny?
**Option A (recommended):** Allow by default. Only paths in `cannot_write_paths` are denied. Everything else is allowed.  
**Option B:** Deny by default. Only paths in `can_write_paths` are allowed. Everything else is denied.

**CTO recommendation:** Option A. Reason: Y* Bridge Labs is a startup, not a locked-down enterprise. Velocity > paranoia. We can tighten later.

**Board override:** If Board prefers Option B for governance purity, CTO will implement deny-by-default. Tradeoff = more YAML maintenance.

### Decision 2: CAPABILITY_REMINDER interval = 50 tool calls or session-based?
**Option A (recommended):** Every N tool calls (N=50). Ensures agent sees reminder even in long single-turn sessions.  
**Option B:** Every session start only (Category 1.5 in boot pack, no mid-session reminders).

**CTO recommendation:** Option A. Reason: Long autonomous sessions (e.g., overnight work) need mid-session nudges.

**Board override:** If Board finds mid-session reminders noisy, switch to Option B.

### Decision 3: Escape hatches = minimal or comprehensive?
**Option A (recommended):** Comprehensive. Every deny category gets explicit escape hatch (e.g., `write_denied` → `default_actions.write_denied`).  
**Option B:** Minimal. Only document universal fallback (`escalate to Board`), let agents figure out workarounds.

**CTO recommendation:** Option A. Reason: This amendment's entire thesis = make envelope visible. Comprehensive escape hatches = fulfillment of that thesis.

**Board override:** If Board wants agents to show more initiative, Option B forces creative problem-solving. Tradeoff = more 6-lock incidents.
